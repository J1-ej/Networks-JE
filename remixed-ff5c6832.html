01
Модель OSI — 7 уровней сети
✓
▶

Модель OSI (Open Systems Interconnection) — стандарт описания того, как данные передаются по сети. Состоит из 7 уровней. Каждый уровень отвечает за свою задачу и «общается» только со смежными уровнями. Твой Python-код работает на уровне 7, а физический кабель — на уровне 1.

#НазваниеПротоколыУстройстваЧто делает
7ПрикладнойHTTP, FTP, DNS, SMTPБраузер, серверИнтерфейс для приложений
6ПредставленияSSL/TLS, JPEG, JSON—Шифрование, кодирование
5СеансовыйNetBIOS, RPC—Управление сессиями
4ТранспортныйTCP, UDP—Надёжная доставка
3СетевойIP, ICMP, ARPРоутерМаршрутизация пакетов
2КанальныйEthernet, Wi-FiКоммутаторПередача кадров
1Физический—Кабель, Wi-FiБиты в сигнал
Браузер → [7 HTTP] → [6 TLS] → [5 Сессия] → [4 TCP] → [3 IP] → [2 Ethernet] → [1 Кабель] ↓ Сервер → [7 HTTP] ← [6 TLS] ← [5 Сессия] ← [4 TCP] ← [3 IP] ← [2 Ethernet] ← [1 Кабель]
💡 Мнемоника

«Привет Представлению Сети, Транспортный Сетевой Канал — Физика!» → П-П-С-Т-С-К-Ф (7→1). Или снизу: «Физики Каналят Сети Только Серьёзно Понимая Приложения».

Задача дня

Показать ответ

а) ping (ICMP) → Уровень 3 (Сетевой)
б) HTTPS → Уровень 7 (Прикладной) + 6 (Представления — TLS)
в) Wi-Fi адаптер → Уровни 1–2 (Физический + Канальный)
г) Flask-сервер → Уровень 7 (Прикладной)
д) HTTP-куки → Уровень 7 (часть HTTP-протокола)

02
TCP/IP и инкапсуляция данных
✓
▶

TCP/IP — реальный стек протоколов интернета (не теоретический как OSI). Состоит из 4 уровней. Инкапсуляция: каждый уровень добавляет свой заголовок к данным. На приёмной стороне заголовки снимаются обратно — декапсуляция.

TCP/IP уровеньСоответствие OSIПротоколыЕдиница данных
Прикладной5–6–7HTTP, FTP, DNS, SMTPСообщение
Транспортный4TCP, UDPСегмент
Интернет3IP, ICMPПакет
Доступа к сети1–2Ethernet, Wi-FiКадр
Данные HTTP: "GET /index.html HTTP/1.1" ↓ +TCP заголовок Сегмент TCP: [порт отпр: 54321 | порт назн: 80 | seq | данные] ↓ +IP заголовок Пакет IP: [IP src: 192.168.1.5 | IP dst: 93.184.216.34 | сегмент] ↓ +Ethernet заголовок Кадр Ethernet: [MAC src | MAC dst | пакет | CRC]
⚠ Важно

TCP/IP и OSI — разные вещи. OSI — теоретическая эталонная модель. TCP/IP — реальный стек который используется в интернете. Знать оба нужно для собеседований.

Задача дня

Нарисуй схему прохождения пакета от твоего браузера до сервера google.com: покажи все уровни, какие заголовки добавляются на каждом, и как они снимаются на стороне сервера.

Показать ответ

Браузер формирует HTTP-запрос → TCP добавляет порты (src≈50000+, dst=80/443) → IP добавляет адреса (твой IP + IP Google) → Ethernet добавляет MAC-адреса → физически уходит по кабелю/Wi-Fi → роутер провайдера снимает Ethernet, смотрит IP, добавляет новый Ethernet-кадр → ... → сервер Google получает кадр, снимает все заголовки, обрабатывает HTTP-запрос.

03
IP-адресация: IPv4, публичные и локальные
✓
▶

IPv4-адрес — 32 бита, записываются как 4 числа от 0 до 255 через точку. Всего ~4,3 млрд адресов. Делятся на классы, публичные (видны в интернете) и приватные (только внутри сети).

04
Подсети, маски и CIDR-нотация
✓
▶

Маска подсети определяет какая часть IP-адреса — сеть, а какая — хост. CIDR (Classless Inter-Domain Routing) — современная запись: 192.168.1.0/24 означает 24 бита под сеть, 8 под хосты = 254 хоста.

CIDRМаскаХостовПрименение
/8255.0.0.016 777 214Крупные ISP
/16255.255.0.065 534Крупные компании
/24255.255.255.0254Малый офис, дом
/30255.255.255.2522Связь двух роутеров
/32255.255.255.2551Конкретный хост
import ipaddress net = ipaddress.ip_network("192.168.1.0/24", strict=False) print(f"Сеть: {net.network_address}") # 192.168.1.0 print(f"Маска: {net.netmask}") # 255.255.255.0 print(f"Broadcast: {net.broadcast_address}") # 192.168.1.255 print(f"Хостов: {net.num_addresses - 2}") # 254 # Все IP в сети for ip in list(net.hosts())[:5]: print(ip)

05
ICMP: ping, трассировка, диагностика
✓
▶

ICMP (Internet Control Message Protocol) — протокол сообщений об ошибках и диагностики. Работает на уровне 3. ping отправляет Echo Request и ждёт Echo Reply. Если ответа нет — хост недоступен или блокирует ICMP.

# Linux / Mac ping -c 4 google.com # 4 пакета ping -i 0.5 -s 1000 8.8.8.8 # интервал 0.5с, размер 1000б traceroute google.com # путь до цели mtr google.com # live-трассировка # Windows ping -n 4 google.com tracert google.com # Python: запуск ping через subprocess import subprocess, platform def ping(host): flag = "-n" if platform.system() == "Windows" else "-c" result = subprocess.run(["ping", flag, "3", host], capture_output=True, text=True) return result.returncode == 0 for host in ["google.com", "8.8.8.8", "192.168.1.1"]: status = "✅ онлайн" if ping(host) else "❌ недоступен" print(f"{host}: {status}")
⚠ Для специалиста ИБ

Многие файрволы блокируют ICMP. Если ping не работает — хост может быть всё равно онлайн. Проверяй через TCP-порты. Ещё: SYN-scan nmap не требует ICMP и часто проходит там, где ping заблокирован.

Задача дня

Напиши Python-скрипт который пингует 5 разных хостов (google.com, ya.ru, 8.8.8.8, 1.1.1.1, любой казахстанский сайт) и выводит отчёт: хост, статус, время ответа в мс.

Показать ответ

06
Traceroute — путь пакета через интернет
✓
▶

Traceroute показывает каждый роутер (хоп) на пути пакета. Использует поле TTL (Time To Live) в IP-заголовке: каждый роутер уменьшает TTL на 1, при TTL=0 отправляет ICMP «Time Exceeded» обратно. Так traceroute узнаёт каждый шаг.

Как работает TTL: Пакет TTL=1 → Роутер1 (TTL=0) → ICMP "Time Exceeded" → мы видим Роутер1 Пакет TTL=2 → Роутер1 → Роутер2 (TTL=0) → ICMP → видим Роутер2 Пакет TTL=3 → Роутер1 → Роутер2 → Роутер3 (TTL=0) → ... ... Пакет TTL=N → ... → Цель (ICMP Echo Reply) → путь пройден!
# Пример вывода traceroute до google.com из Алматы # traceroute google.com 1 192.168.1.1 1.2 ms # твой домашний роутер 2 10.10.1.1 8.3 ms # роутер провайдера 3 72.14.215.165 35.1 ms # Google (Алматы CDN) 4 142.250.79.100 36.8 ms # Google-сервер # Python: парсинг traceroute import subprocess result = subprocess.run(["traceroute","-n","-m","15","google.com"], capture_output=True, text=True) print(result.stdout)

07
ARP и MAC-адреса: как найти устройство в локальной сети
✓
▶

ARP (Address Resolution Protocol) — протокол второго уровня OSI. Решает задачу: у нас есть IP-адрес устройства в локальной сети, но нужен MAC-адрес для отправки кадра. ARP рассылает broadcast-запрос «кто имеет этот IP?» и получает MAC в ответ. Понимание ARP критично для диагностики сети и ИБ.

# ARP-кеш — таблица IP → MAC на твоей машине $ arp -a # все записи в кеше $ arp -n # без резолва имён (быстрее) # Пример вывода: # ? (192.168.1.1) at aa:bb:cc:dd:ee:ff [ether] on eth0 ← роутер # ? (192.168.1.5) at 11:22:33:44:55:66 [ether] on eth0 ← другой ПК # Как работает ARP: # 1. Твой ПК: "Кто имеет 192.168.1.1?" → широковещательный MAC ff:ff:ff:ff:ff:ff # 2. Роутер: "Это я! Мой MAC: aa:bb:cc:dd:ee:ff" → юникаст обратно # 3. Твой ПК записывает в ARP-кеш и отправляет данные # ip neigh — современная альтернатива arp -a $ ip neigh show # 192.168.1.1 dev eth0 lladdr aa:bb:cc:dd:ee:ff REACHABLE # MAC-адрес: структура # aa:bb:cc : dd:ee:ff # └── OUI (производитель) ─┘ └── уникальный номер ──┘ # По OUI можно узнать производителя: maclookup.app # Python: получить свой MAC import uuid, re mac = ':'.join(re.findall('..', '%012x' % uuid.getnode())) print(f"Мой MAC: {mac}") # ARP-спуфинг (атака): злоумышленник рассылает фиктивные ARP-ответы # "Роутер 192.168.1.1 — это я!" → весь трафик идёт через атакующего # Защита: Dynamic ARP Inspection на управляемых коммутаторах
🔒 ARP-спуфинг — классика атак MITM

Именно через ARP-спуфинг работают многие атаки «человек посередине» в локальных сетях. Утилита arpspoof (пакет dsniff) позволяет перехватить трафик в локалке. Защита — статические ARP-записи или DAI на коммутаторе.

08
DNS: как домен превращается в IP
✓
▶

DNS (Domain Name System) — «телефонная книга» интернета. Переводит доменные имена в IP-адреса. Работает иерархически: Root DNS → TLD DNS (.com, .ru) → Authoritative DNS. Ответы кэшируются по TTL.

Процесс разрешения google.com: Браузер → Кэш браузера → Кэш ОС → Resolver (8.8.8.8) ↓ Root DNS → кто отвечает за .com? ↓ TLD .com DNS → кто отвечает за google.com? ↓ Google DNS → google.com = 142.250.74.46
# Терминал: диагностика DNS nslookup google.com # простой запрос dig google.com # подробный ответ dig google.com MX # почтовые серверы dig +trace google.com # полная цепочка # Python: DNS-запрос import socket ip = socket.gethostbyname("google.com") print(f"google.com → {ip}") # Все IP (CDN часто имеет несколько) ips = socket.getaddrinfo("google.com", 80) for info in ips: print(info[4][0])
🔒 ИБ: DNS-спуфинг

Атака «DNS spoofing» подменяет DNS-ответ: жертва вводит bank.com, а DNS возвращает IP злоумышленника. Защита: DNSSEC (подписи записей), DoH (DNS over HTTPS), DoT (DNS over TLS).

Задача дня

Показать ответ

HTTP: структура запроса и ответа
✓
▶

HTTP (HyperText Transfer Protocol) — протокол прикладного уровня для передачи данных. Текстовый (до HTTP/2). Состоит из: стартовой строки, заголовков и тела. Клиент делает запрос — сервер отвечает.

HTTP Запрос: GET /api/users?page=1 HTTP/1.1 Host: api.example.com Authorization: Bearer eyJhbGciOiJIUzI1NiJ9... Content-Type: application/json User-Agent: Mozilla/5.0 (Linux; Android 13) Accept-Encoding: gzip, deflate, br HTTP Ответ: HTTP/1.1 200 OK Content-Type: application/json; charset=utf-8 Content-Length: 342 Cache-Control: max-age=3600 X-RateLimit-Remaining: 98 {"users": [{"id": 1, "name": "Алишер"}]}
💡 Связь с backend

Когда ты пишешь Flask/FastAPI-роут — ты программируешь именно обработку HTTP-запроса. Параметры URL → query string, данные формы → body, авторизация → заголовок Authorization.

HTTPS, SSL/TLS и сертификаты
✓
▶

HTTPS = HTTP + TLS-шифрование. TLS (Transport Layer Security) — протокол шифрования. Защищает от: перехвата трафика (MITM), подмены сервера, изменения данных. Использует асимметричное шифрование для обмена ключами, симметричное — для данных.

TLS Handshake (упрощённо): Клиент → ClientHello (версии TLS, cipher suites) → Сервер Сервер → ServerHello + Сертификат (публичный ключ) → Клиент Клиент → Проверяет сертификат у CA (Let's Encrypt, DigiCert) Клиент → PreMasterSecret (зашифровано публичным ключом) → Сервер Оба → вычисляют SessionKey (симметричный) → Дальше все данные шифруются симметрично (AES-256)
# Python: проверка SSL-сертификата import ssl, socket from datetime import datetime def check_ssl(hostname): ctx = ssl.create_default_context() with ctx.wrap_socket(socket.socket(), server_hostname=hostname) as s: s.connect((hostname, 443)) cert = s.getpeercert() expires = datetime.strptime(cert["notAfter"], "%b %d %H:%M:%S %Y %Z") days = (expires - datetime.now()).days print(f"{hostname}: истекает {expires.date()} (через {days} дней)") print(f"Выдан: {cert['issuer']}") check_ssl("google.com") check_ssl("github.com")
🔒 Атака MITM

При перехвате трафика злоумышленник ставит себя между клиентом и сервером. HSTS (HTTP Strict Transport Security) защищает: браузер всегда использует HTTPS и отклоняет поддельные сертификаты.

✓

HTTP-заголовки безопасности — первая линия обороны веб-приложения. Они инструктируют браузер: какой контент разрешён, куда можно делать запросы, как кешировать. # Ключевые заголовки безопасности: # HSTS — браузер всегда использует HTTPS, даже если ввели http:// Strict-Transport-Security: max-age=31536000; includeSubDomains; preload # CSP — разрешает только указанные источники скриптов/стилей Content-Security-Policy: default-src 'self'; script-src 'self' cdn.js.com # Запрет встройки страницы в iframe (защита от кликджекинга) X-Frame-Options: DENY # Тип контента не угадывается браузером (защита от MIME-sniffing) X-Content-Type-Options: nosniff # Управление реферером — сколько информации передаём о своём URL Referrer-Policy: strict-origin-when-cross-origin # Проверить заголовки сайта через curl: $ curl -I https://github.com $ curl -sI https://google.com | grep -i "strict\|frame\|content-sec" # CORS — заголовки для cross-origin запросов # Сервер говорит: "разрешаю запросы с этих доменов" Access-Control-Allow-Origin: https://myapp.com # конкретный домен Access-Control-Allow-Origin: * # все (опасно для API с авторизацией!) NAT: как роутер подменяет адреса
✓
▶

NAT (Network Address Translation) — механизм который позволяет тысячам устройств с приватными IP-адресами выходить в интернет через один публичный IP. Роутер подменяет адреса на лету и ведёт таблицу трансляции. Именно поэтому у тебя дома адрес 192.168.x.x, а в интернете ты виден как один IP провайдера.

# Типы приватных адресов (RFC 1918) — не маршрутизируются в интернете # 10.0.0.0/8 → крупные корпоративные сети # 172.16.0.0/12 → средние сети # 192.168.0.0/16 → домашние роутеры # Как работает NAT (пример): # Твой ПК: 192.168.1.5:54321 → роутер → 203.0.113.1:41000 → google.com:443 # Ответ: google.com:443 → 203.0.113.1:41000 → роутер → 192.168.1.5:54321 # Роутер знает что порт 41000 принадлежит 192.168.1.5:54321 — таблица NAT # Узнать свой внешний IP (что видит интернет) $ curl ifconfig.me $ curl ipinfo.io/json # Разница: локальный vs внешний $ ip addr show # локальный IP (192.168.x.x) $ curl ifconfig.me # внешний IP (публичный) # Проброс портов (Port Forwarding) — обратное NAT # Задача: сделать твой локальный сервер доступным из интернета # Роутер: "Всё на порт 8080 → перенаправить на 192.168.1.5:80" # Настраивается в веб-интерфейсе роутера: LAN → Port Forwarding # Python: определить свой тип NAT (полезно для P2P) import requests ext_ip = requests.get("https://api.ipify.org").text print(f"Внешний IP: {ext_ip}") # Типы NAT по строгости: # Full Cone NAT → самый открытый, легко P2P # Restricted NAT → только ответы на исходящие # Symmetric NAT → самый строгий, P2P проблематично (Skype, WebRTC используют STUN)
💡 IPv6 и будущее NAT

IPv6 даёт каждому устройству уникальный глобальный адрес — NAT теоретически не нужен. Но NAT стал и защитой: устройства за NAT по умолчанию недоступны снаружи. При переходе на IPv6 нужен firewall вместо NAT.

VPN: как работает туннель
✓
▶

VPN (Virtual Private Network) создаёт зашифрованный туннель между двумя точками. Всё что ты отправляешь — зашифровано и видно только VPN-серверу, а не провайдеру. VPN используют для: безопасного доступа к корпоративной сети, обхода гео-блокировок, защиты в публичных Wi-Fi.

# Как работает VPN-туннель: # Без VPN: Твой ПК → (открытый трафик) → Провайдер → Интернет # С VPN: Твой ПК → (зашифровано) → VPN-сервер → (расшифровано) → Интернет # Провайдер видит только: ты → VPN-сервер. Содержимое — нет. # Популярные протоколы VPN: # OpenVPN → открытый, надёжный, порт 1194/UDP или 443/TCP # WireGuard → новый, очень быстрый, порт 51820/UDP # IPSec/IKEv2 → корпоративный стандарт # PPTP → устаревший, небезопасный — не использовать # WireGuard — установка на Linux $ sudo apt install wireguard $ wg genkey | tee privatekey | wg pubkey > publickey # генерация ключей $ cat privatekey # приватный — никому! $ cat publickey # публичный — отдаём серверу # Конфиг клиента /etc/wireguard/wg0.conf [Interface] PrivateKey = <твой приватный ключ> Address = 10.0.0.2/24 DNS = 1.1.1.1 [Peer] PublicKey = <публичный ключ сервера> Endpoint = vpn.server.com:51820 AllowedIPs = 0.0.0.0/0 # весь трафик через VPN $ sudo wg-quick up wg0 # запустить $ sudo wg-quick down wg0 # остановить $ sudo wg show # статус туннеля # Проверить что VPN работает $ curl ifconfig.me # должен показать IP VPN-сервера
💡 Split tunneling

Split tunneling — режим VPN когда только часть трафика идёт через туннель. Например, корпоративные ресурсы — через VPN, YouTube — напрямую. Настраивается через AllowedIPs в WireGuard: вместо 0.0.0.0/0 указываешь конкретные подсети.

Сниффинг сети: tcpdump без Wireshark
✓
▶

tcpdump — консольный анализатор трафика, доступен на любом Linux-сервере где Wireshark не установишь. Незаменим при отладке сетевых проблем в продакшне: посмотреть реальные пакеты, проверить что запрос дошёл, убедиться что фаервол не режет трафик.

# Базовый захват $ sudo tcpdump -i eth0 # весь трафик на интерфейсе $ sudo tcpdump -i any # все интерфейсы $ sudo tcpdump -i eth0 -n # без резолва имён (быстрее) $ sudo tcpdump -i eth0 -c 100 # захватить 100 пакетов и выйти # Фильтры (BPF — Berkeley Packet Filter) $ sudo tcpdump host 8.8.8.8 # трафик к/от конкретного хоста $ sudo tcpdump port 80 # HTTP-трафик $ sudo tcpdump port 443 # HTTPS (зашифрован — видны только метаданные) $ sudo tcpdump tcp # только TCP $ sudo tcpdump udp port 53 # DNS-запросы # Комбинированные фильтры $ sudo tcpdump host 1.2.3.4 and port 80 $ sudo tcpdump port 80 or port 443 $ sudo tcpdump not port 22 # весь трафик кроме SSH # Сохранить в файл для анализа в Wireshark $ sudo tcpdump -i eth0 -w capture.pcap $ sudo tcpdump -r capture.pcap # читать из файла # Показать содержимое пакетов (ASCII) $ sudo tcpdump -i eth0 port 80 -A # текстовое содержимое $ sudo tcpdump -i eth0 port 80 -X # hex + ASCII # Диагностика DNS $ sudo tcpdump udp port 53 -n -v # наблюдать DNS-запросы в реальном времени # В другом терминале: dig google.com → видишь UDP-пакет в tcpdump
⚠ Только на своих системах

Захват чужого трафика без разрешения — уголовное преступление. tcpdump используется только для диагностики своих серверов и сетей, в учебных средах (VM, localhost) или с явного разрешения владельца.

TCP Handshake и анализ соединений
✓
▶

В Wireshark ты видишь каждый этап установки TCP-соединения. Это важно для понимания атак типа SYN-flood и отладки проблем с подключением.

TCP Handshake в деталях: [SYN] Клиент → Сервер: "Хочу подключиться, мой seq=1000" [SYN-ACK] Сервер → Клиент: "ОК, мой seq=5000, подтверждаю твой ack=1001" [ACK] Клиент → Сервер: "Подтверждаю твой ack=5001" → Соединение открыто Закрытие соединения (FIN-ACK): [FIN] Клиент → Сервер: "Хочу закрыть" [ACK] Сервер → Клиент: "ОК" [FIN] Сервер → Клиент: "Я тоже закрываю" [ACK] Клиент → Сервер: "ОК"
Задача дня

В Wireshark используй фильтр tcp.flags.syn == 1. Найди полный handshake (SYN → SYN-ACK → ACK) для любого соединения. Нажми правую кнопку → «Follow TCP Stream» — увидишь весь разговор клиента и сервера.

Анализ HTTP в Wireshark: запросы и ответы
✓
▶

HTTP (незашифрованный) в Wireshark виден целиком — заголовки, тело, куки. Это наглядно объясняет почему HTTPS обязателен. HTTPS-трафик зашифрован, но метаданные (IP, порты, размер пакетов, SNI — имя домена) видны. Умение читать HTTP в Wireshark — ключевой навык отладки и ИБ.

# Фильтры Wireshark для HTTP http -- все HTTP пакеты http.request -- только запросы http.response -- только ответы http.request.method == "POST" -- только POST http.response.code == 404 -- все 404 http.host == "example.com" -- трафик к конкретному хосту # Анализ запроса в Wireshark — что видно в панели деталей: # ▶ Hypertext Transfer Protocol # GET /api/users HTTP/1.1\r\n # Host: api.example.com\r\n # Cookie: session=abc123\r\n ← видно в открытом HTTP! # Authorization: Bearer token\r\n ← тоже видно! # Follow HTTP Stream — читать весь разговор # ПКМ на пакет → Follow → HTTP Stream # Красный = клиент → сервер # Синий = сервер → клиент # Экспорт объектов — скачать файлы из трафика # File → Export Objects → HTTP # Wireshark покажет список картинок, JS, JSON из перехваченного трафика # HTTPS — что видно и что нет: tls -- TLS handshake пакеты tls.handshake.type == 1 -- ClientHello # В ClientHello видно: SNI (имя домена), версия TLS, cipher suites # НЕ видно: заголовки, тело, куки, токены
🔒 Почему HTTP = катастрофа

В публичном Wi-Fi любой может запустить Wireshark и читать все твои HTTP-запросы: логины, пароли, токены сессий. Один http.request.method == "POST" фильтр — и видны все формы. Всегда HTTPS.

Задача дня

Firewall: принципы блокировки трафика
✓
▶

Firewall (межсетевой экран) фильтрует трафик по правилам: IP-адрес, порт, протокол, направление. Linux: iptables или современный ufw. Принцип «запрещено всё, что не разрешено».

# UFW (Uncomplicated Firewall) — Ubuntu sudo ufw enable # включить sudo ufw status verbose # текущие правила sudo ufw allow 22 # разрешить SSH sudo ufw allow 80/tcp # разрешить HTTP sudo ufw allow 443/tcp # разрешить HTTPS sudo ufw deny 3306 # запретить MySQL извне sudo ufw allow from 192.168.1.0/24 to any port 5432 # PostgreSQL только из локалки # Базовые правила безопасного сервера: # 1. Заблокируй всё по умолчанию sudo ufw default deny incoming sudo ufw default allow outgoing # 2. Открой только нужные порты sudo ufw allow 22 443 80
💡 Для backend-разработчика

Базы данных (PostgreSQL :5432, MySQL :3306, Redis :6379, MongoDB :27017) никогда не должны быть открыты в интернет. Только localhost или внутренняя сеть. Это одна из самых частых ошибок, которая приводит к утечкам данных.

Задача дня

Напиши 5 правил UFW для защиты типичного веб-сервера: открыть только SSH (22), HTTP (80), HTTPS (443). Запретить доступ к PostgreSQL из интернета. Разрешить PostgreSQL только с IP 10.0.0.5 (условный сервер приложений).

Proxy и reverse proxy: nginx как привратник
✓
▶

Прямой прокси (forward proxy) стоит между клиентами и интернетом — скрывает клиентов. Обратный прокси (reverse proxy) стоит перед серверами — скрывает серверы. Nginx как reverse proxy — стандарт для production: принимает все запросы, проксирует на backend, отдаёт SSL, сжимает, кеширует.

# Прямой прокси: клиент → прокси → интернет # Клиент говорит прокси куда идти — прокси делает запрос от своего имени # Применения: корпоративный фильтр, кеш, анонимизация # Обратный прокси: интернет → nginx → backend # Клиент не знает что за nginx — видит только один IP # Конфиг nginx как reverse proxy (Flask на порту 5000) server { listen 80; server_name myapp.com; location / { proxy_pass http://127.0.0.1:5000; # куда проксировать proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; # реальный IP клиента proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; } location /static/ { root /var/www/myapp; # статику отдаём сами expires 30d; # кеш браузера } } # SSL-терминация: nginx расшифровывает HTTPS, передаёт HTTP на backend server { listen 443 ssl; ssl_certificate /etc/letsencrypt/live/myapp.com/fullchain.pem; ssl_certificate_key /etc/letsencrypt/live/myapp.com/privkey.pem; ... } # Load balancing: nginx распределяет запросы между несколькими backend upstream backend { server 127.0.0.1:5001; server 127.0.0.1:5002; server 127.0.0.1:5003; } location / { proxy_pass http://backend; }
💡 Зачем reverse proxy backend-разработчику

Твоё Flask/Django/FastAPI приложение не должно напрямую слушать порт 80/443 — это задача nginx. Nginx отдаёт статику быстрее, управляет SSL, защищает от медленных клиентов (slow-read атаки), делает rate limiting. Это стандартная связка в production.

Задача дня

Установи nginx на своём сервере/VM. Создай минимальное Flask-приложение (hello world). Настрой nginx как reverse proxy: запросы на 80 → Flask на 5000. Проверь что X-Real-IP передаётся корректно. Добавь обслуживание статики через nginx напрямую.

Сетевые атаки: MITM, спуфинг, перехват
✓
▶

MITM (Man-In-The-Middle) — класс атак когда злоумышленник встаёт между двумя общающимися сторонами. Классика в локальных сетях через ARP-спуфинг. DNS-спуфинг перенаправляет на поддельные сайты. IP-спуфинг подделывает адрес отправителя. Знание атак необходимо для понимания защит.

# Виды атак и защита от каждой: ┌─────────────────┬─────────────────────────────┬──────────────────────────────┐ │ Атака │ Механизм │ Защита │ ├─────────────────┼─────────────────────────────┼──────────────────────────────┤ │ ARP Spoofing │ Фиктивные ARP-ответы │ DAI на коммутаторе, HTTPS │ │ DNS Spoofing │ Подмена DNS-ответов │ DNSSEC, DoH (DNS over HTTPS) │ │ SSL Stripping │ Принудительный HTTP │ HSTS, preloading │ │ IP Spoofing │ Подмена src IP в пакете │ BCP38, RPF на роутере │ │ Session Hijack │ Кража куки сессии │ Secure/HttpOnly flags, HTTPS │ └─────────────────┴─────────────────────────────┴──────────────────────────────┘ # Проверить защищён ли сайт от SSL Stripping $ curl -I http://google.com # должен вернуть 301 → HTTPS $ curl -sI https://google.com | grep -i hsts # strict-transport-security: max-age=31536000 ← HSTS активен # DNSSEC — проверить что домен подписан $ dig +dnssec google.com # в ответе будет RRSIG $ delv @8.8.8.8 google.com # полная верификация DNSSEC # DNS over HTTPS — защита DNS от перехвата # Запрос идёт зашифрованным HTTPS на 1.1.1.1 или 8.8.8.8 # Провайдер не видит какие домены ты запрашиваешь # Проверить куки на защитные флаги (в браузере: DevTools → Application → Cookies) # Secure — кука передаётся только по HTTPS # HttpOnly — недоступна из JavaScript (защита от XSS) # SameSite — защита от CSRF
⚠ Только образовательно

Эти атаки рассматриваются исключительно для понимания как защититься. Выполнять MITM-атаки на чужих сетях и устройствах — уголовное преступление. Учебные эксперименты — только в изолированных VM или с явного согласия владельца.

Задача дня

Проверь 5 сайтов на наличие HSTS. Есть ли у банков Казахстана HSTS preloading? Проверь куки через DevTools — есть ли Secure и HttpOnly флаги? Проверь один сайт на DNSSEC через dig +dnssec. Составь таблицу: что защищено, что нет.

DoS и DDoS атаки: теория и защита
✓
▶

DoS (Denial of Service) — атака направленная на недоступность сервиса путём перегрузки ресурсов. DDoS — то же но с тысяч устройств (ботнет). SYN-flood: отправка миллионов SYN-пакетов без завершения handshake, исчерпывает ресурсы сервера.

SYN-flood атака: Атакующий (ботнет) → [SYN] → Сервер создаёт half-open соединение Атакующий (ботнет) → [SYN] → Сервер создаёт ещё... Атакующий (ботнет) → [SYN] → ещё... ещё... ещё... → Сервер исчерпал очередь соединений → легитимные запросы отклоняются Защита SYN-cookies (решение): Сервер не хранит состояние до получения ACK Вместо этого кодирует информацию в SYN-ACK (sequence number) Если ACK приходит — декодирует и создаёт соединение → Полуоткрытые соединения не занимают память

CDN: как ускорить доставку контента
✓
▶

CDN (Content Delivery Network) — сеть серверов по всему миру. Пользователь получает контент с ближайшего к нему сервера, а не с основного. Это сокращает задержку, разгружает origin-сервер и защищает от DDoS. Cloudflare, AWS CloudFront, Fastly — крупнейшие CDN провайдеры.

# Как CDN ускоряет загрузку: # Без CDN: Алматы → сервер в Лондоне → 250ms latency # С CDN: Алматы → CDN-узел в Алмате или Москве → 20ms latency # Что кешируется в CDN: # ✅ Статика: картинки, JS, CSS, шрифты, PDF # ✅ Публичные API-ответы (с Cache-Control: public, max-age=3600) # ❌ Персонализированные ответы, куки, авторизованные запросы # HTTP-заголовки управляют кешированием Cache-Control: public, max-age=86400 -- CDN кешировать 24 часа Cache-Control: private, no-cache -- CDN не кешировать Cache-Control: s-maxage=3600 -- для CDN (shared cache) # Диагностика: определить CDN по заголовкам $ curl -sI https://github.com | grep -i "cf-\|x-cache\|via\|age" # X-Cache: HIT ← контент из кеша CDN # CF-Cache-Status: HIT ← Cloudflare кеш # Age: 3600 ← сколько секунд в кеше # Узнать через какой CDN идёт сайт $ dig github.com # видим IP CDN, не оригинального сервера $ curl -sI https://github.com | grep server # server: cloudflare ← CDN провайдер # Cloudflare дополнительно защищает от DDoS: # Фильтрует трафик прежде чем он доходит до твоего сервера # Rate limiting, WAF (Web Application Firewall), bot protection # Python: проверка заголовков кеша import requests r = requests.get("https://github.com") print(r.headers.get("CF-Cache-Status", "No Cloudflare")) print(r.headers.get("Cache-Control", "No cache policy"))
💡 Cloudflare бесплатный план

Cloudflare Free tier даёт CDN + DDoS-защиту + SSL — бесплатно. Для небольшого проекта это мгновенное ускорение и защита. Просто меняешь NS-записи домена на Cloudflare — и трафик идёт через них.

Итог курса: путь от браузера до сервера
✓
▶

За миллисекунды между нажатием Enter и появлением страницы происходят десятки сетевых операций. Теперь ты знаешь каждую из них.

1. DNS-резолвинг browser cache → OS cache → /etc/hosts → Recursive DNS → Root NS → TLD NS → Auth NS → IP получен, кешируется по TTL 2. TCP-соединение (3-way handshake) SYN → SYN-ACK → ACK → Соединение установлено 3. TLS Handshake (если HTTPS) ClientHello → ServerHello → Certificate → Key Exchange → Finished → Шифрованный канал установлен 4. HTTP-запрос GET / HTTP/1.1 Host: example.com → Через CDN → Firewall → nginx (reverse proxy) → Backend 5. HTTP-ответ 200 OK + Headers + HTML → Браузер рендерит, загружает CSS/JS/картинки (дополнительные запросы) 6. Закрытие соединения FIN → FIN-ACK → ACK (или Keep-Alive для переиспользования)
